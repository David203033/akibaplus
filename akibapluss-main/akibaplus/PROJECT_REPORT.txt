1. INTRODUCTION
====================================================================
AkibaPlus Backend is a web application built using Spring Boot (Java 21) intended to support a SACCOS (Savings and Credit Cooperative Society) management platform. It provides both:
- Server-rendered web pages using Thymeleaf for Admin and Member dashboards.
- REST-style API endpoints used by dashboard JavaScript for interactive features (shares operations, loan actions, meeting attendance, analytics data, etc.).

The application manages core cooperative features such as member profiles, deposits and withdrawals, shares purchase/sale, loans (including eligibility checks), meetings with attendance tracking (including geolocation radius checks), penalties/fines, and notification broadcasting.

The codebase indicates a “backend + UI” monolith approach: the UI templates live inside the same Spring Boot project and are served directly.

====================================================================
2. PROJECT OBJECTIVES AND SCOPE
====================================================================
2.1 Primary Objectives
- Provide a secure membership system with role-based access control.
- Provide admin capabilities to manage members and record transactions.
- Provide member self-service operations: view balances, apply for loans, buy/sell shares, attend meetings, update profile.
- Provide dashboards with analytics charts and summaries.

2.2 In-Scope Features (Based on Source Code)
- User registration and authentication (form login + JWT token generation endpoints).
- Member dashboard: balances, transactions, charts, profile and more.
- Admin dashboard: member lists, savings/loans/shares stats, meetings scheduling, notification sending, expenses recording.
- Data persistence using Spring Data JPA.

2.3 Out-of-Scope / Partially Implemented
Some items are present as placeholders or partially implemented:
- Full JWT authentication filter usage is not present; JWT is generated but the main login mechanism is form-based.
- Some service methods in `AdminService` are stubs.
- Some frontend scripts contain simulated login behavior.
- Flyway migration exists but does not match current JPA entity mappings (tables/column names differ).

5. HIGH-LEVEL ARCHITECTURE
====================================================================
The application follows a standard layered Spring architecture:
- Presentation/UI:
  - Thymeleaf templates: `src/main/resources/templates`
  - Static JS/CSS: `src/main/resources/static`
- Web Controllers:
  - `@Controller` for HTML page rendering
  - `@RestController` for JSON APIs
- Services:
  - Business logic and aggregation (MemberService, AdminService, MeetingService)
- Persistence:
  - Spring Data JPA repositories
- Domain Model:
  - JPA entity classes: User, Member, Loan, Transaction, Meeting, Notification, ShareSale

Data Flow Summary:
1) Browser requests page (e.g., `/dashboard`).
2) Controller loads data from `MemberService` (which calls repositories).
3) Controller injects data into the model for Thymeleaf.
4) The page loads JS/CSS assets.
5) JS calls JSON endpoints to update charts or perform actions (e.g., buy shares).
6) REST controllers update database via repositories/services and return JSON responses.

9. WEB LAYER (CONTROLLERS)
====================================================================
The controller layer contains both `@Controller` (HTML views) and `@RestController` (JSON APIs).

9.1 LandingController
- Endpoint:
  - `GET /` -> returns Thymeleaf template `landing`.

9.2 AuthPageController (HTML)
- Endpoints:
  - `GET /login` -> `templates/auth/login.html`
  - `GET /register` -> `templates/auth/register.html`

9.3 AuthController (JSON)
Base path: `/auth`
- `POST /auth/login`
  - Verifies email+password using `PasswordEncoder.matches`.
  - Generates JWT token via `JwtUtil.generateToken`.
  - Returns `{ token: "..." }`.
- `POST /auth/register`
  - Creates User and linked Member record.
  - Sets default role `ROLE_MEMBER`.
  - Generates membership number `MBR<timestamp>`.
  - Returns `{ message: "Registered" }` or `{ error: "..." }`.

Observations:
- Form login is also configured in SecurityConfig, so the system supports both token issuance and session-based auth.

9.4 MemberDashboardController (HTML)
Base path: `/dashboard`
- `GET /dashboard`
  - Builds the member dashboard model:
    - profile, balances, loan list, transactions, notifications, meetings, penalties, statements.
  - Returns `member` template.
- `GET /dashboard/analytics-data`
  - Returns JSON summary from `memberService.getAnalyticsSummary`.

9.5 AdminDashboardController (HTML)
Base path: `/admin`
- `GET /admin` redirects to `/admin/dashboard`.
- `GET /admin/dashboard` returns `admin` template populated with stats, lists, chart inputs.
- Several JSON helper endpoints exist under `/admin/api/...` inside the same controller for dynamic UI updates.

9.6 MemberApiController (JSON)
Base path: `/api`
Core features:
- Shares:
  - `POST /api/shares/buy`
  - `POST /api/shares/sell`
  - `POST /api/shares/buy-market`
- Loans:
  - `POST /api/loans/apply` (multipart form)
  - `POST /api/loans/pay`
- Savings:
  - `POST /api/savings/deposit`
  - `POST /api/savings/withdraw`
- Penalties:
  - `POST /api/penalties/pay`
- Profile:
  - `POST /api/profile/update` (multipart profile image)
  - `POST /api/profile/password`
- Meetings attendance:
  - `POST /api/meetings/attend` (geolocation + radius)
- Analytics:
  - `GET /api/member/analytics`
- Meetings list:
  - `GET /api/meetings/upcoming`

Notes:
- Uses `Authentication authentication` to get logged-in user email.
- Mixes direct repository access with service calls.

9.7 LoanDataController (JSON)

17. CORE BUSINESS MODULES AND WORKFLOWS
====================================================================
This section describes the main workflows end-to-end.

17.1 Authentication and Registration
A) Registration
- UI: `/register` template.
- JS: `Auth.register()` sends JSON to `/auth/register`.
- Backend:
  - Creates User with encoded password.
  - Creates Member linked to User.

B) Login
- UI: `/login` form posts to `/login`.
- Spring Security authenticates using CustomUserDetailsService.
- Success handler redirects based on role.

JWT Login (API)
- REST endpoint `/auth/login` issues JWT token.
- Not clearly wired into request authentication.

17.2 Member Dashboard
- Entry: `GET /dashboard`
- Data assembled by MemberDashboardController from MemberService.
- Charts:
  - `app.js` calls `GET /api/member/analytics`.

17.3 Admin Dashboard
- Entry: `GET /admin/dashboard`
- AdminDashboardController builds extensive model attributes.
- Charts driven by model-provided `akibaData` variables and Chart.js.

17.4 Savings
- Deposit:
  - `POST /api/savings/deposit` -> calls MemberService.deposit
- Withdraw:
  - `POST /api/savings/withdraw`
  - rule: cannot withdraw until Nov 30 of current year.

17.5 Shares
- Buy shares deducts savings and increases shares value.
- Sell shares creates ShareSale “market entry” and deducts shares value.
- Buy market shares transfers value between buyer and seller.

17.6 Loans
- Eligibility check in MemberService.
- Apply loan:
  - `POST /api/loans/apply` (multipart) creates loan and records disbursement transaction.
- Pay loan:
  - `POST /api/loans/pay` reduces loan balance.
- Loan details:
  - `GET /api/loans/details` provides schedule/derived values.

17.7 Meetings and Attendance
- Admin schedules meeting via `POST /admin/api/meetings/schedule`.
- Member attends meeting via `POST /api/meetings/attend` with geolocation:
  - checks meeting time window
  - checks distance <= radius
  - may create late fine transaction

17.8 Penalties/Fines
- Fines recorded as transactions type `FAINI`.
- Some fines are PENDING and then deducted later (processMeetingPenalties).

17.9 Notifications
- Admin sends notification via `POST /admin/api/notifications/send`.
- Stored in Notification table.
- MemberService currently shows all notifications to members.

18. DATA MODEL AND DATABASE SCHEMA
====================================================================
18.1 JPA vs Migration Script Mismatch
Flyway migration defines tables `users`, `member`, `transaction`.
JPA entities define tables:
- `users`
- `members`
- `transactions`
- `loans`
- `share_sales`

This mismatch suggests:
- Migrations may be outdated or unused.
- With `ddl-auto=update`, Hibernate will create/alter tables to match entities.

18.2 Recommended Schema Alignment
- Either enable Flyway and update migration scripts to match entity tables,
  OR keep Hibernate management but remove/ignore Flyway scripts.
